Git（分散式版本控制工具）

Git的學習是不依賴我們前面學習的知識，就算沒有學習Java也可以學習。

Git就是一個類似於百度雲端硬碟的倉庫。

重點是要掌握使用IDEA操作Git，企業用的最多，一般不會去使用命令。

Git透過不斷階段保存檔案來實現版本控制，比如備份、版本還原等等

每修改一次就認為是一個版本，這個版本是誰寫的，我們加上版本號

Git記錄了開發的全過程，誰在什麼時間做了什麼事情都可以看得很清楚。

1. 目標

  • 了解Git基本概念
  • 能夠概述Git工作流程
  • 能夠使用Git常用命令
  • 熟悉Git程式碼託管服務
  • 能夠使用IDEA操作Git

2. 概述

2.1 開發中的實際場景

2.2 版本控制器的方式

a、集中式版本控制工具
集中式版本控制工具，版本庫是集中存放在中央伺服器的，團隊裡每個人工作時從中央伺服器下載程式碼，是必須聯網才能工作，區域網路或網際網路。個人修改後然後提交到中央版本庫。
舉例：SVN和CVS

b、分散式版本控制工具
分散式版本控制系統沒有「中央伺服器」，每個人的電腦上都是一個完整的版本庫，這樣工作的時候，無需聯網了，因為版本庫就在你自己的電腦上。多人協作只需要各自的修改推送給對方，就能互相看到對方的修改了。
舉例：Git

集中式版本控制的缺點：
需要聯網，中央伺服器磁碟損壞，專案會徹底崩潰。

2.3 SVN

SVN是早期大多數公司在用的版本控制器。

2.4 Git

重點

Git是一個分散式版本管理系統，他透過共享版本庫來共享版本資訊，所以相當於每個開發人員的本地都有一個共享版本庫的副本，所有人員的本地版本庫和共享版本庫都是同步的，所以不用擔心共享版本庫宕機的問題，只要拿一個開發人員的本地版本庫傳到共享版本庫就好了，開發人員之間可以直接交換版本資訊，但是這種方式不常用，我們一般還是透過共享版本庫實現共享，這樣所有的人員都可以共享到版本資訊，無需聯網了，因為版本庫就在你自己的電腦上，但是要實現版本共享的時候還是需要聯網的，自己開發的時候不需要聯網。

2.5 Git工作流程圖

命令如下：

  1. clone（克隆）：從遠端倉庫中克隆程式碼到本地倉庫
  2. checkout（檢出）：從本地倉庫中檢出一個倉庫分支然後進行修訂
  3. add（新增）：在提交前先將程式碼提交到暫存區
  4. commit（提交）：提交到本地倉庫。本地倉庫中儲存修改的各個歷史版本
  5. fetch（抓取）：從遠端庫，抓取到本地倉庫，不進行任何的合併動作，一般操作比較少。
  6. pull（拉取）：從遠端庫拉到本地庫，自動進行合併（merge），然後放到工作區，相當於fetch+merge
  7. push（推送）：修改完成後，需要和團隊成員共享程式碼時，將程式碼推送到遠端倉庫

3.3 基礎操作指令

Git工作目錄下對於檔案的修改（新增、刪除、更新）會存在幾個狀態，這些修改的狀態會隨著我們執行Git的命令而發生變化。

工作區，就是平時存放專案程式碼的地方。

本章節主要講解如何使用命令來控制這些狀態之間的轉換：

  1. git add（工作區 –> 暫存區）git add .新增所有檔案、資料夾和子資料夾，包括.gitignore和以點開頭的任何其他內容；
  2. git commit（暫存區 –> 本地倉庫）

3.3.1 檢視修改的狀態（status）

  • 作用：檢視修改的狀態（暫存區、工作區）
  • 命令形式：git status

新建立的檔案是未追蹤狀態

3.3.2 新增工作區到暫存區（add）

  • 作用：新增工作區一個或多個檔案的修改到暫存區
  • 命令形式：git add 單個檔名|萬用字元
    • 將所有修改加入暫存區：git add .

3.3.3 提交暫存區到本地倉庫（commit）

  • 作用：提交暫存區彙總所有內容到本地倉庫的當前分支
  • 命令形式：git commit -m ‘註解內容’

提交時候新增的備註會被放到日誌中

3.3.4 檢視提交日誌（log）

      • pretty=oneline 將提交資訊顯示為一行

檢視log我們一般都是會加上上面全部的參數的，這樣顯示更美觀有序，我們在上面給這個指令設定了別名

3.4 分支

幾乎所有的版本控制系統都以某種形式支援分支。使用分支意味著你可以把你的工作從開發主線上分離開來進行重大的Bug修改、開發新的功能，以免影響開發主線。master是我們的主線

每個人開發的那一部分就是一個分支，使得每個人的開發互不影響，在每個人都開發完後就將所有的程式碼彙總到一起，此時就要執行分支的合併操作

工作區只能在一個分支工作，每個分支存放的檔案或者資源是不一樣的，就相當於不同的資料夾

head指向誰，誰就是當前分支

3.4.1 檢視本地分支

  • 命令：git branch

*號表示所在的分支

3.4.2 建立本地分支

  • 命令：git branch 分支名
  • 建立的新分支會建立在當前分支的版本之上，所以新建的分支會有當前分支的內容

3.4.3 切換分支（checkout）

  • 命令：git checkout 分支名

我們還可以直接切換到一個不存在的分支（建立並切換）

  • 命令：git checkout -b 分支名

3.4.4 合併分支（merge）

注意：分支上的內容必須先提交，才能切換分支

一個分支上的提交可以合併到另一個分支

  • 命令：git merge 分支名稱
  • 在每個人都開發完後就將所有的程式碼彙總到一起，此時就要執行分支的合併操作

master是我們的主線，我們一般將分支合併到主線上面去

步驟：切換到master分支，然後執行合併命令，執行完後，分支上的資源、檔案就會被合併到主線上面去

例如將test01分支合併到master分支上，master分支上就會加上test01分支上的內容

3.4.5 刪除分支

不能刪除當前分支，只能刪除其他分支

  • git branch -d b1 刪除分支時，需要做各種檢查
  • git branch -D b1 不做任何檢查，強制刪除

小d刪除了就使用D，一般使用d就夠了

我們去刪除沒有合併的分支的時候就會出現刪除不了的情況，此時就可以使用D

3.4.7 開發中分支使用原則與流程

幾乎所有的版本控制系統都以某種形式支援分支。使用分支意味著你可以把你的工作從開發主線上分離開來進行重大的Bug修改、開發新的功能，以免影響開發主線。

在開發中，一般有如下分支使用原則與流程：

  • master（生產）分支
    線上分支，主分支，中小規模專案作為線上執行的應用程式對應的分支；

  • develop（開發）分支
    是從master建立的分支，一般作為開發部門的主要開發分支，如果沒有其他並行開發不同期上線要求，都可以在此版本進行開發，階段開發完成後，需要是合併到master分支，準備上線。
    例如我們要開發新功能，我們可以在develop分支上再建一個分支，新功能一般叫做feature分支，開發完以後再合併到develop分支上面去，而不是直接提交到master分支，最後專案做完了develop再合併到master分支上

develop和master分支是不可刪除的

  • feature/xxxx分支（用完可刪）
    從develop建立的分支，一般是同期並行開發，但不同期上線時建立的分支，分支上的研發任務完成後合併到develop分支，用完後可刪除。

  • hotfix/xxxx分支
    從master派生的分支，一般作為線上bug修復使用，修復測試完成後需要合併到master、test、develop分支。

  • 還有一些其他分支，在此不再詳述，例如test分支（用於程式碼測試）、pre分支（預上線分支）等等。

4. Git遠端倉庫

4.1 常用的託管服務[遠端倉庫]

前面我們已經知道了Git中存在兩種型別的倉庫，即本地倉庫和遠端倉庫。那麼我們如何搭建Git遠端倉庫呢？我們可以借助網際網路上提供的一些程式碼託管服務來實現，其中比較常用的有GitHub、碼雲、GitLab等。

  • GitHub（地址：https://github.com/）是一個面向開源及私有軟體專案的託管平臺，因為只支援Git作為唯一的版本庫格式進行託管，故名GitHub

  • 碼雲（地址：https://gitee.com/）是國內的一個程式碼託管平臺，由於伺服器在國內，所以相比於GitHub，碼雲速度會更快

  • GitLab（地址：https://about.gitlab.com/）是一個用於倉庫管理系統的開源專案，使用Git作為程式碼管理工具，並在此基礎上搭建起來的Web服務，一般用於在企業、學校等內部網路搭建Git私服。

企業裡面我們一般使用GitLab，畢竟程式碼放在自己的機房裡面才安全，GitLab要自己部署

4.5 操作遠端倉庫

4.5.1 新增遠端倉庫

此操作是先初始化本地庫，然後與已建立的遠端庫進行對接。

我們要將本地倉庫和遠端倉庫連接起來，一般一個本地倉庫對應一個遠端倉庫，遠端倉庫預設名為origin

  • 命令：git remote add <遠端名稱> <倉庫路徑SSH>
    • 遠端名稱，預設是origin，取決於遠端伺服器設定
    • 倉庫路徑，從遠端伺服器取得此SSH
    • 例如：git remote add origin git@gitee.com:czbk_zhang_meng/git_test.git

4.5.2 檢視遠端倉庫

  • 命令：git remote

4.5.3 推送到遠端倉庫

  • 命令：git push [-f] [–set-upstream] [遠端名稱 [本地分支名][:遠端分支名]]

    • 如果遠端分支名和本地分支名稱相同，則可以只寫本地分支
      本來是：git push origin master:master 表示將本地倉庫的master分支提交到遠端倉庫的master分支

      • git push origin master 這裡表示將本地倉庫當前master分支的內容推到遠端倉庫上面去

    • -f 表示強制覆蓋

    • –set-upstream 推送到遠端的同時並且建立起和遠端分支的關聯關係。

      • git push –set-upstream origin master

    • 如果當前分支已經和遠端分支關聯，則可以省略分支名和遠端名。

      • git push 將master分支推送到已關聯的遠端分支。

    • [-f] 表示強制覆蓋遠端倉庫的內容

4.5.4 本地分支與遠端分支的關聯關係

  • 檢視關聯關係我們可以使用 git branch -vv 命令

4.5.5 從遠端倉庫克隆

如果已經有一個遠端倉庫，我們可以直接clone到本地。

  • 命令：git clone <倉庫路徑> [本地目錄]
    • 本地目錄可以省略，會自動生成一個目錄，就是SSH後面那部分

不同的主機都把修改完了版本資源放在遠端倉庫上，然後其他人都是克隆，這樣就可以實現不同主機之間的資料同步了，資料都是一樣的

克隆一般只會執行一次，就是在你進去公司的時候，別人提交了以後，我們不需要去克隆整個倉庫，倉庫是很大的，克隆也需要花很多時間，所以要去遠端倉庫中抓取我們要的版本資訊，就是那些別人新增加的提交

4.5.6 從遠端倉庫中抓取和拉取

遠端分支和本地的分支一樣，我們可以進行merge操作，只是需要先把遠端倉庫裡的更新都下載到本地，再進行操作。

  • 抓取命令：git fetch [remote name] [branch name]

    • 抓取指令就是將倉庫裡的更新都抓取到本地，不會進行合併，不合併本地倉庫就是沒有更新，此時還沒有拿到遠端倉庫的內容，合併後才會拿到更新的內容
    • 如果不指定遠端名稱和分支名，則抓取所有分支。

  • 拉取命令：git pull [remote name] [branch name]

    • 拉取指令就是將遠端倉庫的修改拉到本地並自動進行合併，等同於fetch+merge
    • 如果不指定遠端名稱和分支名，則抓取所有並更新當前分支。

只去執行抓取命令git fetch，本地倉庫是沒有遠端倉庫來的內容的
